# 第一阶段：项目需求边界与总体架构

## 1. 项目名称

基于 Yjs 的支持隐私保护的结构化数据协同编辑器

## 2. 项目背景

实时协同编辑系统允许多个用户同时编辑同一份共享数据。CRDT（Conflict-free Replicated Data Types，无冲突复制数据类型）能够在没有全局锁和中心化操作顺序的情况下合并并发更新，保证多副本最终一致，因此被广泛应用于协同编辑场景。

然而，传统 CRDT 协同编辑系统通常默认所有参与者同步同一份完整数据视图。对于包含隐私信息、权限分级信息或部门隔离信息的结构化数据，这种设计会带来隐私泄露风险。低权限用户即使在界面上看不到某些内容，也可能因为同步了完整 CRDT 文档而在本地获得无权访问的数据。

本项目针对这一问题，在 Yjs 提供的完整 CRDT 数据层之上设计隐私视图变换层。系统根据用户身份、角色和节点属性生成用户专属视图，并将用户在视图中的编辑操作经过逆向映射和权限校验后写回完整文档，从而在多用户协同编辑场景下同时实现数据一致性与访问隔离。

## 3. 项目目标

本项目旨在实现一个结构化数据协同编辑器原型，满足以下目标：

1. 使用 Yjs 作为底层 CRDT 一致性引擎，管理完整 JSON 树文档。
2. 支持多用户对结构化数据进行协同编辑。
3. 通过 RBAC + ABAC 混合访问控制策略，为不同用户生成不同的隐私视图。
4. 用户客户端只接收其有权限访问的视图数据，不直接同步完整 Yjs 文档。
5. 用户在视图中的编辑操作需要经过逆向映射和权限校验，才能写回完整文档。
6. 在保证访问隔离的同时，保留 CRDT 的并发合并和最终一致性能力。

## 4. 核心问题

本项目重点解决传统 CRDT 协同编辑中的以下问题：

传统 CRDT 系统通常假设所有用户拥有同一份完整数据副本。虽然这种方式有利于实现最终一致性，但不适用于存在权限差异的协同编辑场景。例如，在项目文档树、组织任务列表或员工信息表中，不同用户应当只能访问与自己权限相关的数据子集。

因此，本项目的核心问题是：

> 如何在复用 Yjs 作为底层 CRDT 一致性引擎的前提下，使不同权限用户只能看到和编辑自己的数据视图，同时保证所有合法编辑最终能够合并到同一份完整文档中？

## 5. 系统边界

第一版系统采用客户端-服务器架构。

服务端被视为可信节点，负责保存完整 Yjs 文档、执行隐私视图变换、校验用户权限并合并合法操作。普通用户客户端不直接同步完整 Y.Doc，而是通过服务端获得根据权限生成的用户视图。

### 5.1 系统内部负责的内容

- 完整 JSON 树文档的 CRDT 存储与更新。
- 用户、角色、节点属性和访问控制策略管理。
- 根据访问控制策略生成用户专属视图。
- 将用户视图操作逆向映射为完整文档操作。
- 对写操作进行权限校验。
- 将合法操作写入 Yjs 完整文档。
- 将完整文档变化重新投影为各用户视图。
- 提供基础 Web 编辑界面和 WebSocket 通信能力。

### 5.2 第一版暂不负责的内容

- 不实现完全去中心化 P2P 架构。
- 不让低权限用户直接同步完整 Y.Doc。
- 不实现复杂富文本编辑器。
- 不实现端到端加密或安全多方计算。
- 不实现完整通用 BX 语言，仅实现面向 JSON 树的可解释投影与逆向操作映射。
- 不处理企业级身份认证系统，只实现课程项目所需的简单用户与角色配置。

## 6. 最小可行功能

### 6.1 数据结构

系统第一版支持 JSON 树结构。每个节点包含唯一 ID、节点名称、节点内容、节点属性、权限元数据和子节点列表。

示例结构如下：

```json
{
  "id": "node-1",
  "title": "项目文档",
  "content": "根节点内容",
  "attrs": {
    "department": "dev",
    "confidentialLevel": 1,
    "ownerId": "u1"
  },
  "children": []
}
```

### 6.2 基本编辑操作

第一版至少支持以下操作：

- 添加子节点。
- 删除节点。
- 重命名节点。
- 修改节点内容。
- 修改节点属性。

移动节点可作为扩展功能，在核心流程稳定后实现。

### 6.3 隐私策略

系统至少实现两类隐私策略。

第一类是基于角色的访问控制，即 RBAC。系统根据用户角色判断其可见范围和可编辑范围。例如：

- `admin` 可以查看和编辑完整文档。
- `manager` 可以查看部门内大部分节点，但不能查看高密级字段。
- `member` 只能查看与自己相关或本部门公开的节点。
- `guest` 只能查看公开节点。

第二类是基于属性的访问控制，即 ABAC。系统根据节点属性和用户属性共同判断访问权限。例如：

- 用户所属部门必须与节点 `department` 相同。
- 用户权限等级必须大于等于节点 `confidentialLevel`。
- 用户 ID 必须等于节点 `ownerId`，或者用户具有管理角色。

### 6.4 协同能力

系统需要支持多用户同时编辑。所有合法操作最终写入服务端完整 Yjs 文档，并由 Yjs 处理并发合并。

对于普通用户，系统不直接广播完整文档变化，而是根据每个用户的权限重新生成对应视图，或生成视图层面的增量更新。

## 7. 总体架构

系统采用分层架构：

```text
用户浏览器
  |
  | 视图请求 / 视图编辑操作
  v
Web 前端编辑器
  |
  | WebSocket / HTTP
  v
服务端 API 与同步模块
  |
  | 用户身份、操作请求
  v
访问控制层 RBAC + ABAC
  |
  | 可见性判断、可编辑性判断
  v
隐私视图变换层
  |
  | getView / putOperation
  v
Yjs 完整 CRDT 文档层
  |
  | 合并、持久化、广播
  v
文档存储与测试数据
```

## 8. 核心模块划分

### 8.1 CRDT 完整数据层

该层基于 Yjs 实现完整 JSON 树文档的存储和更新。完整文档只存在于服务端或管理员上下文中。

主要职责：

- 创建和维护 `Y.Doc`。
- 使用 `Y.Map` 和 `Y.Array` 表示 JSON 树。
- 提供节点增删改查接口。
- 接收合法操作并写入 Yjs 文档。
- 负责并发编辑下的最终一致性。

### 8.2 访问控制层

该层实现 RBAC + ABAC 混合访问控制策略。

主要职责：

- 管理用户、角色和权限等级。
- 根据用户信息和节点属性判断节点是否可见。
- 判断用户是否可以对节点执行指定操作。
- 在用户操作写回完整文档前进行权限校验。

### 8.3 隐私视图变换层

该层是本项目的核心创新部分，位于完整 CRDT 文档和用户视图之间。

主要职责：

- 正向变换：`getView(fullDoc, user)`，将完整文档投影为用户专属视图。
- 逆向映射：`putOperation(viewOperation, user, fullDoc)`，将用户视图操作转换为完整文档操作。
- 保证用户无法通过视图操作修改无权访问的数据。
- 在完整文档发生变化后，为不同用户重新计算或增量更新视图。

### 8.4 服务端同步层

该层负责前端与完整文档之间的通信。

主要职责：

- 建立 WebSocket 连接。
- 根据用户身份下发初始视图。
- 接收用户视图操作。
- 调用访问控制层和视图变换层。
- 将合法操作提交给 Yjs 文档。
- 将文档变化投影后推送给相关用户。

### 8.5 Web 前端编辑器

该层负责展示用户可见的结构化树，并提供基础编辑能力。

主要职责：

- 展示当前用户的视图树。
- 支持添加、删除、重命名、修改内容等操作。
- 将用户操作转换为视图操作发送给服务端。
- 接收服务端推送的最新视图并更新界面。

## 9. 数据流设计

### 9.1 用户读取视图

```text
用户登录
  -> 服务端读取用户身份和角色
  -> 访问控制层加载策略
  -> 隐私视图变换层执行 getView(fullDoc, user)
  -> 返回用户专属视图
  -> 前端渲染视图树
```

### 9.2 用户提交编辑

```text
用户在前端编辑视图
  -> 前端生成 viewOperation
  -> 服务端接收 viewOperation
  -> 访问控制层检查用户是否有权执行该操作
  -> 隐私视图变换层将 viewOperation 映射为 fullDocOperation
  -> fullDocOperation 写入 Yjs 完整文档
  -> Yjs 合并并触发文档更新
  -> 服务端为各用户重新生成视图并推送
```

### 9.3 并发编辑处理

```text
用户 A 和用户 B 同时编辑各自视图
  -> 两个操作分别经过权限校验
  -> 合法操作被转换到完整文档层
  -> Yjs 对并发操作进行合并
  -> 完整文档达到最终一致
  -> 各用户视图由合并后的完整文档重新投影得到
```

## 11. 第一阶段结论

第一阶段确定系统边界如下：

> 本项目使用 Yjs 管理完整 JSON 树，通过 RBAC + ABAC 访问控制策略生成用户隐私视图，并将用户视图操作逆向映射到完整文档。系统采用客户端-服务器架构实现多人协同编辑。服务端保存完整 CRDT 文档，普通用户客户端只接收自身权限范围内的视图数据，从而解决传统 CRDT 协同编辑中所有用户默认共享完整视图所带来的隐私问题。

在该边界下，后续阶段将重点设计 Yjs 中的完整数据模型、权限策略格式、视图变换算法以及操作逆向映射机制。
