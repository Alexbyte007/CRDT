## 1. 项目概述

### 1.1 项目目标

本项目实现了一个基于 CRDT（Conflict-free Replicated Data Types，无冲突复制数据类型）的隐私保护树形结构文档协同编辑器。系统面向多人协同编辑结构化项目文档、研发任务树、部门资料库等场景，在保证多人编辑最终一致性的同时，解决传统 CRDT 系统“所有客户端同步完整文档副本”所带来的隐私泄露问题。

传统协同编辑系统通常将完整文档同步到每个参与者客户端，再在界面层隐藏用户无权查看的内容。这种方式虽然便于实现实时同步，但对于包含权限分级、部门隔离、项目预算、内部任务说明等敏感数据的结构化文档而言并不安全。低权限用户即使在界面上看不到某些节点，也可能通过本地数据、网络报文或调试工具接触到完整文档内容。

针对上述问题，本项目采用“可信服务端 + 隐私视图投影 + 逆向写回”的方案：服务端保存完整 Yjs 文档，客户端只接收根据用户身份、角色和节点级 ACL 配置生成的用户专属视图；用户在视图中的操作必须经过服务端权限校验和操作映射后，才能写回完整 CRDT 文档。通过这种方式，系统在复用 Yjs 并发合并能力的同时，实现了面向不同用户的访问隔离。

项目具体目标如下：

1. 使用 Yjs 作为底层 CRDT 一致性引擎，维护完整树形结构文档。
2. 支持多用户对同一份结构化文档进行协同编辑。
3. 支持固定三级树形结构：一级项目空间、二级模块或工作包、三级任务节点。
4. 支持节点添加、删除、重命名、内容编辑、属性编辑等基本编辑操作。
5. 通过 RBAC（Role-Based Access Control，基于角色的访问控制）和节点级 ACL（Access Control List，访问控制列表）生成用户隐私视图。
6. 客户端不直接同步完整 Y.Doc，只接收服务端投影后的可见视图。
7. 通过 `getView` 实现完整文档到用户视图的正向变换，通过 `putOperation` 实现用户视图操作到完整文档操作的逆向写回。
8. 支持 WebSocket 实时同步、离线操作队列、断线重连后批量同步。
9. 支持用户注册、登录、会话管理、管理员用户管理和多角色访问演示。
10. 提供自动化测试、类型检查和 Docker 部署配置，保证课程验收和演示的可复现性。

## 2. 需求分析

### 2.1 功能性需求

本项目功能性需求围绕“结构化文档编辑、CRDT 协同一致性、隐私视图保护、多用户同步和工程化演示”展开。

#### 2.1.1 结构化树形文档管理

系统需要支持树形结构文档，文档节点以项目空间、模块或工作包、任务节点的三级结构组织。每个节点包含唯一标识、父节点标识、节点类型、标题、正文内容、业务属性、权限元数据、创建信息和更新时间等字段。

系统应支持以下基础编辑操作：

1. 新增节点：在根节点或指定父节点下添加子节点。
2. 删除节点：删除指定节点，并根据权限和冲突情况处理其子树。
3. 保留子节点删除：删除中间节点并将其子节点提升到合适位置。
4. 重命名节点：修改节点标题。
5. 修改节点内容：修改节点正文，任务内容支持 Markdown 文本。
6. 修改节点属性：修改优先级、预算、任务状态等结构化属性。
7. 修改节点权限：管理员或具备权限的用户可调整节点可见性和角色权限。

#### 2.1.2 CRDT 协同编辑

系统需要支持多个用户同时编辑同一份结构化文档。合法操作写入服务端完整 Yjs 文档后，由 Yjs 负责并发更新合并，从而保证完整文档在并发编辑和网络延迟条件下能够达到最终一致。

项目中的 CRDT 协同需求包括：

1. 完整文档使用 Yjs 数据结构保存。
2. 节点注册表、根节点顺序、子节点顺序和文本内容都应具备可合并能力。
3. 并发修改同一节点内容时，底层文本使用 `Y.Text` 提升文本合并效果。
4. 删除与迟到更新发生冲突时，采用删除优先策略，避免已删除节点被离线操作复活。
5. 服务端应维护 tombstone 墓碑记录，以支持删除冲突处理、Undo/Redo 和后续垃圾回收。

#### 2.1.3 隐私保护视图

系统需要解决“不同用户只能看到自己有权访问的数据”的问题。普通用户客户端不能同步完整 Yjs 文档，而只能获得服务端根据权限策略生成的用户视图。

隐私保护视图需求包括：

1. 根据用户角色、节点可见性、节点级 ACL 和显式授权生成用户专属视图。
2. 无权访问的节点不出现在用户视图中，也不泄露标题、内容、属性和权限元数据。
3. 对于用户可见但字段权限不足的节点，应隐藏或脱敏不可见字段。
4. 不可见中间节点应被折叠，可见后代可提升到最近可见祖先下，避免暴露受限路径。
5. 视图节点保留稳定 `nodeId`，用于后续逆向写回。
6. 视图节点包含权限摘要 `permissions`，用于前端控制按钮可用状态。

#### 2.1.4 双向变换与权限写回

系统需要实现完整文档到用户视图的正向变换，以及用户视图操作到完整文档操作的逆向变换。

正向变换需求：

1. `getView(crdt, user)` 根据完整文档和用户信息生成用户视图。
2. 正向投影必须执行节点级可见性过滤、字段级脱敏和权限摘要生成。
3. 不同用户登录后应看到不同结构和不同字段内容。

逆向写回需求：

1. `putOperation(crdt, user, operation)` 将用户视图操作映射为完整文档操作。
2. 服务端必须重新校验用户身份、节点存在性、节点可见性和操作权限。
3. 用户不能通过伪造请求修改不可见节点或无权编辑字段。
4. 批量结构化更新应逐条映射到真实节点，并逐条校验权限。
5. 操作写回后，服务端应重新计算各在线用户视图并推送。

#### 2.1.5 用户、角色与权限管理

系统需要提供基本账户体系，使不同角色用户能够登录并展示不同隐私视图。

用户管理需求包括：

1. 支持用户注册，注册用户默认角色为访客。
2. 支持密码登录、登出、会话查询和旧会话失效。
3. 支持管理员查看用户列表、修改用户角色、删除用户。
4. 用户角色包括 `admin`、`manager`、`member`、`guest`。
5. 权限策略结合 RBAC 与节点级 ACL：RBAC 根据角色决定基础能力，节点级 ACL 根据节点自身的可见性、允许角色、可编辑角色、显式允许用户和显式拒绝用户进行细粒度控制。

#### 2.1.6 网络同步与离线编辑

系统需要支持在线协同和断线后恢复同步。

网络同步需求包括：

1. HTTP 接口用于登录、视图查询、操作提交、用户管理等请求。
2. WebSocket 用于推送用户专属视图、在线状态和协同变化。
3. 客户端通过 WebSocket Ping/Pong 心跳、连接断开事件和传输异常共同判断连接健康状态。
4. 离线时用户操作进入本地持久化队列，队列项状态包括 `pending`、`sending`、`acked`、`rejected`。
5. 重连后客户端批量提交离线队列，服务端逐条返回 `applied`、`skipped` 或 `rejected`。
6. 服务端需要识别重复操作，避免离线重放造成重复写入。

#### 2.1.7 前端演示与用户交互

系统需要提供可操作的 Web 图形界面，用于课程验收展示。

前端需求包括：

1. 登录页支持不同用户登录和注册。
2. 主页面展示当前用户的树形文档视图。
3. 节点操作按钮根据权限动态启用或禁用。
4. 支持 Markdown 内容编辑与预览。
5. 支持任务属性筛选、聚合统计和批量更新；统计范围仅限当前用户可见节点，避免通过预算总额、平均预算等聚合结果反向推断不可见数据。
6. 支持操作日志、在线人数、节点编辑状态、Undo/Redo、用户资料菜单和设置弹窗。
7. 网络断开时提示离线状态，重连后自动同步队列。

### 2.2 非功能性需求

#### 2.2.1 安全性

系统的安全边界是可信服务端。服务端保存完整文档，客户端只能接收投影视图。所有写操作必须在服务端进行身份认证、权限校验和节点状态校验，防止客户端绕过前端页面直接调用接口修改无权数据。

安全性要求包括：

1. 普通用户客户端不得获取完整 `Y.Doc`。
2. 服务端不得将节点 `acl` 等敏感权限元数据直接暴露给低权限用户。
3. 所有需要登录的接口通过会话令牌访问。
4. 管理员接口仅允许 `admin` 角色访问。
5. 用户被删除或降级后，旧会话应失效。

#### 2.2.2 一致性与可靠性

系统需要保证合法操作最终合并到完整文档中，并在不同用户视图中反映出来。

一致性与可靠性要求包括：

1. 完整文档层依赖 Yjs 保证并发更新的最终一致性。
2. 服务端对离线操作进行重校验，确保权限变化后旧操作不能越权写入。
3. 删除操作采用墓碑记录，避免离线迟到修改复活已删除节点。
4. Undo/Redo 与 tombstone GC 需要协同，避免仍可能被撤销的删除节点被过早清理。
5. 文档、用户和权限数据应支持 SQLite 持久化，服务重启后仍可恢复。

#### 2.2.3 可维护性

项目采用 TypeScript 编写，通过类型定义约束核心数据结构和操作格式。系统按模块划分为访问控制、CRDT 文档、视图变换、服务端、前端页面和测试等部分，降低功能之间的耦合。

可维护性要求包括：

1. 核心类型集中定义，便于统一维护。
2. 权限策略配置化，减少硬编码。
3. `getView`、`putOperation`、`validateViewOperation` 等核心流程可测试、可解释。
4. 自动化测试覆盖关键模块，降低后续修改风险。

#### 2.2.4 可部署性

系统应支持本地运行和容器化部署。开发者可以通过 `npm install`、`npm run typecheck`、`npm test`、`npm start` 进行本地开发与验证，也可以通过 Docker Compose 在 Linux 服务器或云服务器上部署演示。

### 2.3 开发约束

项目开发受到课程时间、题目范围和技术选型等限制，因此系统边界作出如下约束：

1. 系统采用客户端-服务器架构，不实现完全去中心化 P2P 协同。
2. 服务端被视为可信节点，负责保存完整文档、执行权限校验和合并合法操作。
3. 普通用户客户端不直接同步完整 Yjs 文档，不实现端到端加密或安全多方计算。
4. 项目聚焦结构化树形文档编辑，不实现完整富文本编辑器。
5. 权限模型采用课程项目可解释的 RBAC + 节点级 ACL 策略，不接入企业级身份认证系统。
6. 当前版本适合单个 Node.js 服务实例运行；SQLite、内存会话和单进程 WebSocket 不适合直接进行多实例负载均衡。
7. 课程报告与演示以原型系统和关键技术验证为目标，重点展示 CRDT 一致性、隐私视图和逆向写回机制。

## 3. 系统设计

### 3.1 总体设计

系统采用分层架构，由浏览器客户端、服务端接口层、访问控制层、隐私视图变换层、CRDT 完整文档层和持久化层组成。

```text
浏览器客户端
  |
  | HTTP / WebSocket
  v
服务端接口层
  |
  | 用户身份、视图操作、离线队列
  v
访问控制层 RBAC + 节点级 ACL
  |
  | 可见性判断、字段权限、操作权限
  v
隐私视图变换层
  |
  | getView 正向投影 / putOperation 逆向写回
  v
Yjs 完整 CRDT 文档层
  |
  | Y.Map / Y.Array / Y.Text / tombstone
  v
SQLite 持久化与配置文件
```

整体设计思想是将“协同一致性”和“隐私隔离”分层处理。Yjs 负责完整文档内部的并发合并与最终一致性；访问控制和视图变换层负责决定不同用户能看到什么、能改什么；服务端同步层负责将用户操作送入该流程并向各用户广播对应视图。

系统中的关键数据流如下：

1. 用户登录后，服务端根据会话令牌识别用户身份。
2. 服务端调用访问控制层和 `getView`，从完整文档生成用户视图。
3. 浏览器渲染用户视图树，并根据节点权限显示可用操作。
4. 用户在页面中执行编辑操作，前端生成 `ViewOperation`。
5. 服务端接收操作后执行身份认证、权限校验和逆向映射。
6. 合法操作被转换为 `FullDocOperation` 并写入 Yjs 完整文档。
7. 文档变化后，服务端为每个在线用户重新生成隐私视图并通过 WebSocket 推送。
8. 离线期间产生的操作进入本地队列，重连后按队列批量提交并逐条校验。

### 3.2 详细设计

#### 3.2.1 CRDT 完整文档设计

完整文档由服务端维护为一份 Yjs 文档，顶层结构如下：

```text
Y.Doc
  ├── meta: Y.Map
  ├── nodes: Y.Map<NodeId, Y.Map>
  ├── rootIds: Y.Array<NodeId>
  └── tombstones: Y.Map<NodeId, Y.Map>
```

其中，`meta` 存储文档编号、标题、schema 版本和更新时间；`nodes` 是节点注册表，保存所有活跃节点；`rootIds` 保存根节点顺序；`tombstones` 保存已删除节点的墓碑快照。

每个树节点主要字段如下：

| 字段 | 含义 |
| --- | --- |
| `id` | 节点唯一标识，用于视图映射和操作定位。 |
| `parentId` | 父节点标识，根节点为 `null`。 |
| `type` | 节点类型，包括 `folder`、`doc`、`task`。 |
| `title` | 节点标题。 |
| `content` | 节点正文内容，底层可由 `Y.Text` 承载。 |
| `attrs` | 业务属性，如部门、负责人、标签、优先级、预算、任务状态等。 |
| `acl` | 节点权限元数据，如可见性、允许角色、可编辑角色、显式允许用户、拒绝用户等。 |
| `children` | 子节点 ID 列表，保持树形顺序。 |
| `createdBy` / `updatedBy` | 创建者和最后修改者。 |
| `createdAt` / `updatedAt` | 创建时间和最后更新时间。 |

采用“节点注册表 + 父子关系”的扁平化树结构有两个优点：第一，服务端可以通过 `nodeId` 快速定位节点并进行权限校验；第二，视图中的节点可以保留真实 ID，便于将用户操作逆向映射回完整文档。

#### 3.2.2 用户模型与权限模型设计

系统用户模型包括：

```ts
interface User {
  id: string;
  username: string;
  name: string;
  role: "admin" | "manager" | "member" | "guest";
  department: string;
  createdAt: number;
}
```

系统定义四类角色：

| 角色 | 权限定位 |
| --- | --- |
| `admin` | 管理员，具有完整文档查看、编辑、用户管理和权限调整能力。 |
| `manager` | 部门负责人，可查看和编辑部门范围内大部分节点，部分敏感字段受限。 |
| `member` | 普通成员，只能查看本部门或本人相关节点，可进行有限编辑。 |
| `guest` | 访客，只能查看公开内容，默认不能编辑。 |

权限判断采用 RBAC + 节点级 ACL 结合的方式：

1. RBAC 根据用户角色决定基础能力，例如管理员可以管理用户，访客不能编辑节点。
2. 节点级 ACL 根据每个节点自身的访问控制列表进行细粒度判断，例如 `visibility`、`allowedRoles`、`editableRoles`、`contentEditableRoles`、`childAddableRoles`、`deletableRoles`、`allowedUsers` 和 `deniedUsers`。
3. 高级权限字段用于处理特殊操作，例如删除冲突处理权限 `deleteConflictResolverUserIds`。

节点可见性判断可概括为：

```text
canViewNode(user, node) =
  user.role == "admin"
  OR (
    user.id 不在 node.acl.deniedUsers 中
    AND (
      node.acl.visibility == "public"
      OR user.id 在 node.acl.allowedUsers 中
      OR user.role 在 node.acl.allowedRoles 中
    )
  )
```

节点编辑权限在可见性的基础上进一步判断具体操作类型。例如，重命名需要 `canRename`，修改正文需要 `canEditContent`，删除需要 `canDelete`，修改预算、优先级或任务状态需要对应的结构化属性权限。

#### 3.2.3 隐私视图正向变换设计

隐私视图正向变换由 `getView(crdt, user)` 实现，输入为完整 CRDT 文档和用户身份，输出为当前用户可见的 `UserView`。

正向变换流程如下：

```text
读取完整文档快照
  -> 遍历根节点和子树
  -> 对每个节点执行 canViewNode
  -> 过滤不可见节点
  -> 对可见节点执行字段级脱敏
  -> 生成 permissions 权限摘要
  -> 处理不可见中间节点与可见后代提升
  -> 输出用户专属视图
```

用户视图节点结构如下：

```ts
interface ViewNode {
  id: NodeId;
  parentId: NodeId | null;
  type: NodeType;
  title: string;
  content?: string;
  attrs?: Partial<NodeAttrs>;
  children: ViewNode[];
  permissions: ViewPermissions;
}
```

与完整节点相比，视图节点不直接暴露完整 `acl`，只保留前端必要的权限摘要。例如，前端只需要知道当前用户是否可以添加子节点、删除节点、重命名、编辑内容、修改属性，而不需要知道完整权限策略。

正向变换的关键设计是“非同构视图”。当中间节点不可见但其后代节点可见时，系统不会显示“受限路径”虚拟节点，也不会暴露不可见节点标题，而是将可见后代提升到最近可见祖先下。这样既保证用户能够访问合法内容，又避免通过路径结构泄露隐私信息。

#### 3.2.4 逆向写回设计

逆向写回由 `putOperation(crdt, user, viewOperation)` 和服务端操作处理逻辑共同完成。用户视图中的操作不能直接修改完整文档，必须经过服务端重新校验。

逆向写回流程如下：

```text
浏览器生成 ViewOperation
  -> HTTP 或 WebSocket 提交到服务端
  -> 服务端验证会话令牌
  -> 检查操作目标节点是否存在且对用户可见
  -> 检查用户是否具备该操作权限
  -> 将 ViewOperation 转换为 FullDocOperation
  -> 写入 Yjs 完整文档
  -> 持久化并广播新视图
```

逆向写回重点解决三个问题：

1. 防伪造：即使用户绕过前端按钮构造请求，服务端也会重新校验权限。
2. 防越权：用户不能通过传入不可见节点 ID 修改无权节点。
3. 防敏感字段污染：用户提交属性更新时，服务端只接受其有权修改的字段。

例如，普通成员可以更新任务正文，但不能修改节点 ACL；访客即使构造 `updateAcl` 请求，也会被服务端拒绝。批量结构化更新同样不会直接修改临时筛选列表，而是提取真实 `nodeId` 后为每个节点生成独立操作并逐条校验。

#### 3.2.5 删除冲突与 tombstone 设计

协同编辑中常见冲突之一是“某用户离线期间编辑节点，另一用户在线删除该节点”。如果处理不当，迟到的离线修改可能会复活已删除节点。为避免该问题，系统采用删除优先策略和 tombstone 墓碑机制。

删除节点时，系统将被删除节点快照移入 `tombstones`，并从活跃节点集合中移除。后续如果收到针对该节点的迟到更新，服务端会识别该节点已被删除，拒绝或跳过相应操作，不允许其重新出现在活跃树中。

tombstone 还服务于两个功能：

1. Undo/Redo：撤销删除时需要从墓碑中恢复节点快照。
2. 垃圾回收：长期不再被引用的墓碑可以由 tombstone GC 清理，减少存储占用。

为避免破坏撤销能力，GC 会跳过仍被 undo/redo 栈保护的墓碑节点。

#### 3.2.6 服务端接口设计

系统服务端提供 HTTP API 和 WebSocket 两类接口。

主要 HTTP 接口如下：

| 接口 | 方法 | 功能 |
| --- | --- | --- |
| `/health` | GET | 健康检查。 |
| `/api/register` | POST | 用户注册。 |
| `/api/login` | POST | 用户登录并获取会话令牌。 |
| `/api/logout` | POST | 用户登出。 |
| `/api/session` | GET | 查询当前登录会话。 |
| `/api/users` | GET | 管理员查看用户列表。 |
| `/api/users/:userId` | PATCH | 管理员修改用户角色或信息。 |
| `/api/users/:userId` | DELETE | 管理员删除用户。 |
| `/api/view` | GET | 获取当前用户隐私视图。 |
| `/api/operations` | POST | 提交单个视图操作。 |
| `/api/operations/batch` | POST | 批量提交离线操作队列。 |
| `/api/delete-impact` | POST | 删除前分析影响范围。 |
| `/api/undo` | POST | 撤销操作。 |
| `/api/redo` | POST | 重做操作。 |

WebSocket 连接格式为：

```text
ws://localhost:3000/ws?token=<token>
```

WebSocket 主要用于在线视图推送、心跳检测、在线人数和节点编辑状态维护。服务端不会向所有用户广播同一份完整文档，而是对每个在线用户分别生成其权限范围内的隐私视图。

#### 3.2.7 持久化与部署设计

系统支持 SQLite 持久化，保存文档状态、用户账户和权限配置。通过持久化机制，服务重启后能够恢复上一次编辑结果和用户信息。

部署方面，项目提供 `Dockerfile` 和 `docker-compose.yml`，可在本地或云服务器上通过容器方式启动服务。Docker 部署能够统一 Node.js 运行环境、端口映射和数据卷路径，降低课程验收时的环境差异风险。

## 4. 系统实现

### 4.1 技术栈与目录结构

项目主要技术栈如下：

| 类别 | 技术 |
| --- | --- |
| 开发语言 | TypeScript |
| 运行环境 | Node.js |
| CRDT 引擎 | Yjs |
| 网络通信 | HTTP + WebSocket（`ws`） |
| 持久化 | SQLite CLI / 文件数据目录 |
| 测试框架 | Vitest |
| 部署方式 | Docker / Docker Compose |

项目核心目录如下：

```text
src/
  access-control/   权限策略与策略表达式引擎
  crdt/             Yjs 文档封装、节点操作、冲突处理、tombstone GC
  fixtures/         示例文档、内置账号和测试数据
  server/           HTTP、WebSocket、认证、持久化、Undo/Redo、在线状态
  view/             getView / putOperation 隐私视图双向变换

  *.test.ts         单元测试、集成测试和页面逻辑测试
```

### 4.2 CRDT 文档层实现

CRDT 文档层位于 `src/crdt/`。该层封装 Yjs 文档结构，向上提供节点增删改操作，避免上层模块直接操作底层 `Y.Map` 和 `Y.Array`。

主要实现内容包括：

1. `document.ts`：创建和初始化 CRDT 文档，维护文档元信息、节点集合、根节点顺序和墓碑集合。
2. `operations.ts`：实现 `addNode`、`deleteNode`、`deleteNodeKeepChildren`、`renameNode`、`updateContent`、`updateAttrs`、`updateAcl` 等完整文档操作。
3. `snapshot.ts`：将 Yjs 文档转换为普通 JSON 快照，便于视图生成、测试断言和持久化。
4. `conflicts.ts`：处理删除优先、离线迟到操作等协同冲突。
5. `tombstone-gc.ts`：对长期无保护引用的墓碑节点进行垃圾回收。
6. `state-vector.ts`：维护 CRDT 状态向量相关能力，为同步和状态比较提供基础。

项目没有手写复杂的 CRDT 合并算法，而是复用 Yjs 成熟的共享数据类型。这样可以将开发重点放在结构化树文档、隐私投影和权限写回上，同时保证底层协同一致性具备可靠基础。

### 4.3 权限控制层实现

权限控制层位于 `src/access-control/`。该层实现默认权限策略、策略表达式解释和访问判断逻辑。

主要实现内容包括：

1. `policy-types.ts`：定义策略配置的数据结构。
2. `default-policy.ts`：提供默认 RBAC + 节点级 ACL 权限规则。
3. `expression.ts`：实现策略表达式求值，支持基于用户角色、节点 ACL 字段和操作类型的规则判断。
4. `policy-engine.ts`：封装策略判断入口，为视图变换和服务端操作校验提供统一接口。
5. `policy.ts`：提供权限判断辅助函数。

系统将权限判断拆分为“可见性、内容编辑、子节点添加、删除、属性编辑、ACL 编辑”等多个维度。这样前端可以根据 `permissions` 精确控制按钮状态，服务端也可以按操作类型进行严格校验。

### 4.4 隐私视图变换实现

隐私视图变换位于 `src/view/transform.ts`，是本项目的核心模块之一。

正向投影 `getView` 的实现重点包括：

1. 从完整文档快照出发，递归遍历树形结构。
2. 对每个节点执行可见性判断。
3. 对可见节点构造 `ViewNode`，只保留当前用户可访问字段。
4. 为每个视图节点生成权限摘要，如 `canAddChild`、`canDelete`、`canRename`、`canEditContent` 等。
5. 对不可见中间节点执行折叠处理，将可见后代提升到合法位置。

逆向写回 `putOperation` 的实现重点包括：

1. 根据 `ViewOperation` 类型定位目标节点。
2. 检查目标节点是否存在、是否已删除、是否对用户可见。
3. 调用权限引擎判断用户是否具备对应操作权限。
4. 将视图操作转换为完整文档操作。
5. 对新增节点补全安全默认属性和 ACL。
6. 对属性更新执行字段级过滤，避免用户修改无权字段。

该模块体现了双向变换思想：正向变换用于“从完整数据生成安全视图”，逆向变换用于“从安全视图操作恢复完整数据操作”。与简单的前端隐藏不同，所有安全边界都在服务端执行。

### 4.5 服务端实现

服务端位于 `src/server/`，负责将用户请求、权限校验、CRDT 操作和视图推送串联起来。

主要模块如下：

1. `main.ts`：服务启动入口，读取环境变量和持久化配置。
2. `app.ts`：创建 HTTP 服务器和 WebSocket 服务器，初始化上下文、权限引擎、Undo 管理器和在线状态管理器。
3. `http.ts`：处理登录、注册、视图查询、操作提交、批量同步、用户管理等 HTTP 请求。
4. `websocket.ts`：处理 WebSocket 连接、会话认证、视图推送和断开清理。
5. `auth.ts`：实现用户账户、密码登录、会话令牌和会话失效。
6. `operations.ts`：处理视图操作校验、操作去重、完整文档写入和结果返回。
7. `delete-impact.ts`：删除前分析影响范围，识别用户无权删除的后代节点。
8. `undo.ts`：实现按会话隔离的撤销和重做栈。
9. `awareness.ts`：维护在线人数、节点编辑状态和心跳状态。
10. `persistence.ts`：实现 SQLite 持久化读写。

服务端核心上下文包含 CRDT 文档、用户集合、账户集合、会话集合、权限引擎、已处理操作 ID 集合、Undo 管理器和 tombstone 保留时间等信息。每次合法操作写入后，服务端会持久化文档并调用广播逻辑，为所有在线用户分别推送最新隐私视图。

### 4.6 前端页面实现

项目采用服务端内嵌页面方式实现 Web 前端，主要代码位于 `src/server/page.ts`。前端不依赖复杂框架，重点满足课程演示中的核心交互。

前端实现内容包括：

1. 登录、注册和会话状态恢复。
2. 树形视图展示和节点选择。
3. 节点新增、删除、重命名、内容编辑和属性编辑。
4. Markdown 编辑与预览。
5. 任务筛选、聚合统计和批量属性更新。
6. 操作日志展示。
7. Undo/Redo 操作。
8. WebSocket 连接、心跳、重连和离线队列。

### 4.7 离线同步实现

离线同步机制用于处理网络短暂中断或用户离线编辑的场景。客户端在无法连接服务器时，将用户操作保存到本地可靠队列；恢复连接后，批量提交队列中的操作。

离线同步过程如下：

```text
用户离线编辑
  -> 前端生成 ViewOperation
  -> 操作以 pending 状态写入本地持久化队列
  -> 网络恢复后自动重连
  -> 可同步操作进入 sending 状态
  -> 批量提交 /api/operations/batch
  -> 服务端逐条校验身份、权限、节点状态和重复操作
  -> 返回 applied / skipped / rejected
  -> 客户端将成功操作标记为 acked，将失败操作标记为 rejected
  -> 客户端清理过期确认项并刷新视图
```

该机制保证了离线操作不会绕过权限模型。即使用户在离线时有权限编辑，重连时如果该节点已被删除或用户权限已变化，服务端仍会根据当前完整文档状态重新判断并拒绝不合法操作。

### 6.2 技术难点与解决方案

#### 6.2.1 难点一：CRDT 与权限视图的结合

传统 CRDT 协同编辑通常要求每个副本持有完整数据，而本项目需要让不同用户只接收权限范围内的数据。直接将完整 Y.Doc 同步到客户端会导致隐私泄露。

解决方案是采用可信服务端架构。服务端持有完整 Yjs 文档，客户端只接收经过服务端投影的用户视图。这样既保留了 Yjs 在完整文档层的并发合并能力，又避免低权限用户获得完整数据副本。

#### 6.2.2 难点二：用户视图操作如何写回完整文档

不同用户看到的树结构可能不同，特别是不可见中间节点被折叠后，用户视图和完整文档不再同构。如果直接按照视图结构写回，可能破坏完整文档结构或造成越权修改。

解决方案是让视图节点保留真实 `nodeId`，并在服务端通过 `putOperation` 将视图操作映射为完整文档操作。服务端不信任前端状态，而是重新读取完整文档、重新判断权限，再决定是否写入。

#### 6.2.3 难点三：离线操作与删除冲突

用户离线期间可能编辑某个节点，而该节点在线期间被其他用户删除。重连后如果直接应用离线操作，会导致已删除节点被复活。

解决方案是采用删除优先和 tombstone 墓碑机制。删除操作会记录节点快照，后续迟到操作如果指向已删除节点，会被识别并拒绝或跳过。Undo/Redo 可在需要时从墓碑恢复节点，长期无用墓碑则由 GC 清理。

#### 6.2.4 难点四：权限策略既要灵活又要可测试

如果权限规则全部硬编码在业务逻辑中，后续调整困难，也难以解释。课程报告和演示需要清晰说明“为什么某个用户能看见或不能看见某个节点”。

解决方案是将权限逻辑模块化，采用 RBAC + 节点级 ACL 组合模型，并通过策略配置和策略引擎集中执行判断。这样权限规则可以单独测试，也便于在报告和演示中说明。

### 6.3 不足与改进方向

由于课程项目时间有限，当前系统仍存在一些可改进之处：

1. 当前系统采用单服务实例运行，尚未支持多实例部署下的会话共享和 WebSocket 横向扩展。
2. 当前前端为轻量级页面，适合课程演示，但还不是完整产品级前端工程。
3. 权限策略虽然已经支持 RBAC + 节点级 ACL，但还可以进一步扩展为更通用的策略语言和可视化配置界面。
4. 当前没有实现端到端加密，可信边界仍在服务端。
5. SQLite 适合课程演示和小规模部署，后续可扩展为 PostgreSQL 等更适合生产环境的数据库。
6. CRDT 文档的增量视图推送仍可进一步优化，减少频繁重新投影带来的计算开销。
7. 当前尚未接入 SonarQube、CodeQL 等持续代码质量平台，循环依赖和格式规范问题仍需在后续工程化迭代中持续治理。

后续如果继续完善，可以从以下方向展开：

1. 支持多实例部署和 Redis Pub/Sub 跨实例广播。
2. 引入更完整的前端框架和组件化工程。
3. 增加权限策略可视化编辑器。
4. 增加审计日志、操作回放和版本历史对比。
5. 对大型文档进行增量投影和局部更新优化。
6. 引入更强的身份认证机制和 HTTPS 部署。
7. 在 CI 流程中加入 ESLint 或 Biome、CodeQL、SonarCloud、依赖审计和重复代码检查，形成自动化质量门禁。

### 6.4 个人与团队收获

通过本项目，团队成员对协同编辑系统、分布式数据一致性、访问控制模型和工程化开发流程有了更深入理解。项目不仅涉及传统 Web 系统的前后端开发，还涉及 CRDT、隐私视图、双向变换、离线同步和冲突处理等较复杂的系统设计问题。

在研发过程中，团队逐步认识到：协同编辑系统的难点不只是“让多个用户都能编辑”，更重要的是在网络延迟、权限差异、离线操作和冲突删除等复杂情况下，仍然保证数据一致、权限正确、用户体验可接受。通过模块化设计、自动化测试和阶段性集成，项目最终形成了较完整的课程设计成果。

总体而言，本项目达成了预期目标：实现了一个基于 Yjs CRDT 的结构化树形协同编辑器，并在其上设计了面向隐私保护的用户视图投影和逆向写回机制，能够有效展示“CRDT 最终一致性”和“不同权限用户视图隔离”两项核心能力。
